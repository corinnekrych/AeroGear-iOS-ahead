Modularization Proposal


Separate Repos vs Variant

In iOS, modules translate into:

  • separate gihub
  • or use cocoapods subspec


Pod::Spec.new do |s|
s.name         = "AeroGear-OAuth"
s.platform     = :ios, 7.0
s.default_subspec = 'standard'

s.subspec 'common' do |ss|
    ss.source_files = 'AeroGear-iOS/**/*.{h,m}'
    ss.public_header_files = 'AeroGear-iOS/AeroGear-OAuth.h'...
    ss.requires_arc = true
    ss.dependency 'AeroGear-Pipe'
end
s.subspec 'standard' do |ss|
    ss.dependency 'AeroGear-Store'
    ss.dependency 'AeroGear-OAuth/common'
end
s.subspec 'encrypted' do |ss|
   ss.dependency 'AeroGear-Encrypted-Store'
   ss.dependency 'AeroGear-OAuth/common'
end

Core Data (IncrementalStore Plug)



"You might use a custom incremental store to retrieve data from a web service, query an unsupported database format, interface with legacy code, or bridge with cross-platform code bases."


Benefits:

  • Offers cache (upon fault retrieve)
  • Can adapt to different type of remote provider
  • Out-of-the-box UI support (e.g. NSFetchedResultsController)
  • Large CoreData audience

Offer pluggable incremental store types



Offer adapters like in those examples:

  • StackMob Incremental Store
  • SecureIncrementalStore (proprietary)
  • CouchBaseIncrementalStore
  • ZumeroIncrementalStore (against Zumero backend)
  • LiveOakIncrementalStore ??
